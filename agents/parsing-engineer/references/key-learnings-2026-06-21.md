## Cron-промпт должен быть конкретным

**Проблема**: Слишком общий промпт layer3-parse ("запусти парсинг новых источников") приводил к тому что LLM тратил токены на анализ, путался и крашился с RuntimeError.

**Решение**: Конкретный алгоритм с лимитами:
- Максимум 2-3 источника за запуск
- Чёткие шаги: проверить существующие → запустить скрипт → загрузить → отчитаться
- При ошибке — логировать и переходить к следующему
- Формат отчёта: что спарсено, сколько записей, ошибки

## web_search подход — более продуктивен

Для стран с малым количеством данных (TJ, TM, KG):
1. `web_search` по запросам `"страна тип_продукции производитель название"`
2. Извлечь названия компаний из результатов
3. Для каждой компании — дополнительный поиск контактов
4. Скомпилировать в JSON и загрузить в Supabase

**Результат**: 13 таджикских + 4 кыргызских компаний найдены за 10 минут.

## Supabase: source_year поле

- `source_year` (integer) — обязательное поле в raw_parsed_data
- Не включать ключ если значение пустое, или конвертировать в int
- Пустая строка `""` вызывает ошибку `22P02: invalid input syntax for type integer`

## Supabase: count=exact не работает для raw_parsed_data

- Возвращает HTTP 400 с ошибкой `"failed to parse filter (exact)"`
- **Решение**: использовать пагинацию с offset и len() для подсчёта

## Supabase: русские символы в URL query string

- `urllib.request` НЕ кодирует русские символы в query params автоматически
- **ВСЕГДА** использовать `urllib.parse.urlencode()`:
```python
params = urllib.parse.urlencode({'select': 'country', 'country': 'Россия'})
url = f"{SUPABASE_URL}/rest/v1/raw_parsed_data?{params}"
```
