### 3. Найди и исправь критический баг
Симптом: Hermes отправляет запрос → Consilium обрабатывает → провайдер возвращает 200 OK → но Hermes получает "Provider failed after retries".

Проверь:
- Всю цепочку возврата ответа (где теряется?)
- Формат JSON ответа (соответствует ли OpenAI API?)
- Стриминг vs non-streaming (какой реально используется?)
- Таймауты (Hermes ждёт 45с, Consilium — 30с)
- Circuit Breaker (не блокирует ли рабочих провайдеров?)
- Rate Limiter (не отключает ли ключи ложно?)
- System Prompt Filter (не вырезает ли нужное?)

### 4. Проверь полный Flow согласно README.md
Telegram → Hermes → Consilium → Task Router → Fallback Manager → Rate Limiter → Provider → Ответ → Hermes → Telegram

Каждый шаг должен работать как описано в README:
- "Привет" → chat → groq/cloudflare → ответ
- "Напиши код" → code → openrouter → ответ
- При ошибке провайдера → fallback к следующему
- При 429 → cooldown → следующий ключ
- При 401/402/403 → ключ отключается
- Circuit Breaker при 5 сетевых ошибках

### 7. Балльная система провайдеров и моделей
Вместо жёсткого PRIORITY нужна динамическая система:
- Каждый провайдер/модель получает баллы за: успешные запросы, низкую задержку, большой дневной лимит
- Теряет баллы за: 429, 5xx, таймауты, маленький контекст
- При каждом запросе выбирается провайдер с наивысшим баллом
- Баллы обновляются в реальном времени и сохраняются в SQLite
- После перезапуска баллы восстанавливаются

## ПРИНЦИПЫ СИСТЕМЫ (из README.md)
- Модульные провайдеры (каждый в отдельном файле, автообнаружение)
- Единый OpenAI-совместимый API для всех провайдеров
- Provider-agnostic роутинг
- Per-key rate limiting с SQLite-сохранением
- Circuit breaker для сетевых ошибок
- System prompt filter для очистки от технического мусора Hermes
- Шифрование ключей AES-256-GCM
- Адаптивный приоритет на основе статистики успешности
- Dashboard, Alerting, Request ID трассировка
