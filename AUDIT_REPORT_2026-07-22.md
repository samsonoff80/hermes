## 1. Критический баг: найден и исправлен

**Симптом:** провайдер возвращает 200 OK → Hermes показывает
«⚠️ The model provider failed after retries».

### Корневая причина

`tools` не передавались провайдеру. [`consilium_server.py:686`](consilium/consilium_server.py)
читал `tools = body.get("tools", [])`, использовал их только для набора имён в
`rescue_inline_tool_calls`, но в `payload` не клал — параметра `tools`
не было даже в сигнатуре `call_provider`.

### Цепочка отказа

Подтверждена по дампам gateway-слоя Hermes в `logs/`:

```
Hermes шлёт tools[read_file, delegate_task, terminal, ...]
   → Consilium отбрасывает tools
   → модель не знает об инструментах, отвечает обычным текстом
   → агент не может выполнить ни одного действия
   → ядро крутит цикл восстановления: nudge → prefill → empty-retry → fallback
   → исчерпание попыток → result['error']
   → run.py:21052   final_response = f"⚠️ {result['error']}"
   → run.py:463     _looks_like_gateway_provider_error() матчит "error code:" / "http NNN"
   → run.py:398     _gateway_provider_error_reply() подменяет текст
   → Telegram: "The model provider failed after retries"
```

### Ключевое наблюдение

Эта строка **не приходит от провайдера и не является ошибкой транспорта** —
её подставляет `_gateway_provider_error_reply` в `run.py:398`. Подмена
выполняется только на чат-поверхностях:

```python
_GATEWAY_RAW_TEXT_PLATFORMS = frozenset({"local", "api_server", "webhook", "msgraph_webhook"})
```

Для `local` (CLI/TUI) и `api_server` подмены нет — поэтому в терминале баг
выглядел иначе, чем в Telegram, что и сбивало диагностику.

Условие срабатывания именно этой формулировки: текст ≤ 400 символов,
≤ 4 переводов строки, начинается с `API call failed` / `non-retryable error` /
`error code:` / `http NNN` / `invalid api key`, **и при этом** не содержит
маркеров auth / policy / rate-limit (иначе сработали бы другие ветки).

### Вторая, независимая причина того же симптома

Свежий `logs/consilium_latest.log` (22.07, 08:20) показывает состояние, в котором
система **заблокировала сама себя**:

```
[WARNING] ⏳ openrouter:0 rate-limited (cooldown:21519s)   ← ~6 часов
[WARNING] 🔴 groq: circuit breaker blocked                  (все 3 ключа)
[WARNING] 🔴 github: circuit breaker blocked                (все 3 ключа)
[WARNING] ⏳ sambanova:0 rate-limited (disabled)
[WARNING] ⏳ sambanova:1 rate-limited (cooldown:20752s)
[WARNING] ⏳ sambanova:2 rate-limited (cooldown:20850s)
```

Все провайдеры одновременно недоступны → `HTTP 503` → у Hermes это ошибка
вида `Error code: 503` → срабатывает та же подмена текста.

Причины залипания устранены в этом аудите:
- `mark_402` отключал ключ **навсегда**, без возможности восстановления;
- успешный запрос не сбрасывал cooldown (`mark_success` не вызывался);
- circuit breaker с порогом 10 не размыкался обратно при успехе на другом ключе;
- эскалация доходила до 6 часов и там оставалась.

Теперь: успешный ответ сбрасывает `consecutive_429` и cooldown
(`rate_limiter.mark_success`), circuit breaker закрывается при первом успехе,
а заблокированные провайдеры не исчезают из цепочки — уходят в хвост как
последний шанс.

### Третье подтверждение: сломанный SQL статистики

В том же логе на каждый запрос:

```
[WARNING] Stats failed: Incorrect number of bindings supplied.
          The current statement uses 7, and there are 6 supplied.
```

То есть статистика провайдеров **не записывалась вообще** — при том, что
`AUDIT_REPORT.md` от 21.07 отмечал эту ошибку как исправленную. В версии
аудита `provider_stats.py` переписан, SQL заменён на `executemany` с
батчевым сбросом.

### Подтверждение из логов проекта

В `logs/consilium_responses.log` все 5 успешных ответов:

```
✅ mistral/codestral-2508 -> content: 1308 chars, tool_calls: 0 in 2.67s
✅ mistral/codestral-2508 -> content: 1604 chars, tool_calls: 0 in 2.25s
✅ mistral/codestral-2508 -> content: 2257 chars, tool_calls: 0 in 3.38s
✅ mistral/codestral-2508 -> content: 1599 chars, tool_calls: 0 in 2.94s
✅ mistral/codestral-2508 -> content: 1569 chars, tool_calls: 0 in 2.48s
```

`tool_calls: 0` во всех случаях. Агент болтал текстом по 1300–2250 символов
вместо вызова инструментов.

### Исправление и доказательство

Добавлен проброс полей OpenAI-спеки в `call_provider`. Написан интеграционный
тест `consilium/tests/test_tools_passthrough.py`, поднимающий фейкового

```

✅ tool_choice доставлен

### Найдено в параллельной ветке main (PR #4)

Во время аудита в `main` влили независимые правки тех же файлов. Две из них
содержат дефекты — в версии аудита они отсутствуют, файлы переписаны целиком.

**1. Нарушение отступов в `fallback_manager.build_chains()`:**

```python
            for model in p.get("models", []):
                ...
                dps = provider_stats.get_dynamic_score(name) if provider_stats else 0
            all_entries.append({          # ← ВНЕ цикла по моделям
                    "provider": name,
                    "model": model,       # ← значение последней итерации
```

`all_entries.append()` оказался на уровень выше цикла `for model`. Следствия:
- в цепочку попадёт **только последняя модель** каждого провайдера;
- при пустом списке моделей `model`, `tags`, `dps` не определены →
  **`UnboundLocalError`** при старте.

**2. Эскалация cooldown в `rate_limiter.mark_429()`:**

```python
        prev = self._get_consecutive_429(provider, key_index)   # SELECT на каждый 429
        step = min(prev, len(COOLDOWN_STEPS) - 1)
        cooldown = time.time() + COOLDOWN_STEPS[step]
        self._save_state(...)
        logger.warning(f"... cooldown {COOLDOWN_STEPS[0]}s")     # ← всегда 90s в логе
```

Логика верна, но лог всегда печатает `COOLDOWN_STEPS[0]` вместо фактического
шага — при отладке это вводит в заблуждение (в логе «90s», в базе 6 часов).
Плюс отдельный `SELECT` к SQLite на каждый 429.

**Принято из main без изменений:** `load_dotenv` в `providers/__init__.py`
(решает расхождение путей к `.env`), порядок моделей `groq`, правило
`.gitignore` для `CONSILIUM_FIX_REPORT.md`.

## 3. Балльная система DPS

Реализована в [`consilium/provider_stats.py`](consilium/provider_stats.py).
Считается на пару **провайдер/модель**, а не только на провайдера.

### Формула

```
DPS = 40·success + 25·latency + 20·limits + 15·health

success = (ok + 2) / (ok + fail + 4)        байесовское сглаживание
latency = exp(-avg_latency / 2.0)            EMA, свежие замеры весомее (α=0.3)
limits  = min(1, log10(rpd) / 4)             дневной лимит провайдера
health  = 1 - severity · exp(-age / 600)     штраф за сбой тает за ~10 минут
```

`severity` по видам сбоя: `429` → 1.0, `network` → 0.9, `5xx` / `timeout` → 0.8,
прочее → 0.5.

### Почему именно так

**Байесовское сглаживание** решает две реальные проблемы жёсткого
`success/(success+fail)`:
- новичок с результатом 1/1 не обгоняет проверенного провайдера с 900/1000;
- единичный сбой не обнуляет рейтинг стабильного провайдера.

**Затухание health по времени** возвращает провайдера в строй автоматически,
без ручного сброса: свежий 429 бьёт сильно, десятиминутной давности — почти нет.

### Измеренное поведение

| Сценарий | DPS |
|---|---|
| 20 успехов, задержка 0.4 с | **87.1** |
| 20 отказов 5xx | **33.8** |
| быстрый (0.3 с) против медленного (8 с) | **85.8** против **64.7** |
| 100 успехов, затем 1 сбой | 88.7 → **73.3** (не обвал) |
| после перезапуска процесса | **85.3** — восстановлено из SQLite |

### Интеграция

`fallback_manager._rank()` сортирует цепочку по убыванию DPS, тай-брейкеры —
холодный приоритет и количество ключей.

**Жёсткие гейты применяются отдельно от баллов:** `disabled`, активный
`cooldown`, разомкнутый circuit breaker уводят провайдера в **хвост** цепочки,
но не выбрасывают его — если живых не осталось, он остаётся последним шансом.

### Персистентность

Состояние живёт в памяти, сбрасывается в SQLite батчами раз в 30 с и
принудительно при остановке сервиса (`lifespan` shutdown). При старте
восстанавливается. Это устраняет главную проблему прежней реализации —
открытие соединения с БД на каждый запрос.

### Наблюдение

```bash

```

## 4. Совместимость с Hermes v0.19

Проверено по release notes `v2026.7.20` («The Quicksilver Release»,
опубликован 2026-07-20), `cli-config.yaml.example` и официальной документации.

### Важное для прокси

- **Consilium на 127.0.0.1 распознаётся как локальный эндпоинт.** Hermes
  автоматически поднимает socket-read таймаут до 1800 с и отключает
  stale-детекторы. Прежний `request_timeout: 45` не работал в любом случае.
- Hermes при `provider: custom` шлёт топ-левел `reasoning_effort` и
  `think: false`. Прокси со строгой валидацией тела вернул бы 400 —
  Consilium эти поля игнорирует, это корректное поведение.
- **Fallback у Hermes — per-turn, а не per-session.** Каждое новое сообщение
  пользователя возвращает трафик на Consilium, даже если он падал в прошлом ходе.
- `agent.api_max_retries` по умолчанию 3. При наличии fallback-провайдеров
  документация советует ставить 1 для быстрого failover.
- Требования к `finish_reason` в документации **не описаны**, но код Consilium
  всё равно приведён к спецификации: `null` больше не возвращается.
- Системный промпт в v0.19 стал байт-стабильным между ходами ради
  prompt-кэша (PR #67403). Фильтр Consilium детерминирован, кэш не ломает.

### Требования к эндпоинту, подтверждённые документацией

- обязателен `POST /v1/chat/completions`;
- **нативный tool-calling в OpenAI-формате обязателен** — если сервер отдаёт
  вызовы текстом, Hermes считает это поломкой;
- строгое чередование ролей: подряд только `tool`-сообщения;
- `choices[].message.content` не должен быть пустым при отсутствии `tool_calls` —
  именно поэтому добавлен `response_has_payload()`;
- `GET /v1/models` не обязателен, но используется для автодетекта
  `context_length`;
- `usage.prompt_tokens` де-факто нужен: по нему срабатывает порог компрессии
  контекста.

### Окружение

`requires-python = ">=3.11,<3.14"`, `openai==2.24.0` (жёсткий пин),
`httpx[socks]==0.28.1`, `pydantic==2.13.4`. Совпадает с `logs/hermes_version.txt`.

## 5. Состояние агентов

### Общая ошибка протокола (исправлена)

> `delegate_task` **не принимает** параметр `toolsets`.
> Дословно из документации: *«delegate_task does not accept a model-facing
> toolsets parameter. Each subagent inherits the parent's enabled toolsets so
> the model cannot grant a child capabilities that the parent does not have.»*

`toolsets=["file","terminal"]` был прописан в **трёх** местах — исправлено везде:

- `config.yaml` → `agent.system_prompt`
- `agents/orchestrator/SOUL.md`
- `agents/orchestrator/SKILL.md`

### Реальная сигнатура delegate_task в v0.19

| Параметр | Значение |
|---|---|
| `goal` | что сделать (обязательный) |
| `context` | весь контекст — субагент стартует с **чистой историей** |
| `role` | `"leaf"` (по умолчанию) или `"orchestrator"` |
| `max_iterations` | лимит ходов на субагента |
| `tasks` | список задач для батч-вызова |

Leaf-субагенту недоступны: `delegate_task`, `clarify`, `memory`, `execute_code`.

### Важное следствие

Субагент стартует **без SOUL.md** (`skip_context_files`) и с пустой историей.
Принятый в проекте подход — передавать склейку `SOUL + SKILL + PROGRESS`
через `context` — единственно верный. Это зафиксировано в промпте оркестратора.

## 7. Расширяемость

| Вопрос | Ответ |
|---|---|
| **Как добавить провайдера?** | Файл в `consilium/providers/`, наследник `BaseProvider`, затем импорт и запись в `ALL_PROVIDERS` в `providers/__init__.py`. Автообнаружения по каталогу **нет** — заявленное в README «автообнаружение» это ручной список |
| **Как добавить тип задачи?** | Ключ в `TASK_KEYWORDS` (`router.py`) и в `TAG_RULES` (`fallback_manager.py`). Раньше требовалась правка ещё и инлайн-логики сервера |
| **Как изменить приоритет?** | Теперь не нужно: порядок определяет DPS. `COLD_START_PRIORITY` влияет только на холодный старт |
| **Поддерживается ли прокси?** | В README заявлено («если ключ начинается с `proxy:`»), в коде **не реализовано**. `httpx` умеет через `HTTPS_PROXY`, но per-key прокси нет |
| **Как добавить ключ?** | `PREFIX_N` в `.env`. Исправлено: дырки в нумерации больше не обрывают загрузку, читается и `os.environ` |
| **Автообнаружение провайдеров?** | Нет, см. выше |

## 8. Граничные случаи

| Ситуация | Поведение после исправлений |
|---|---|
| Все провайдеры упали | HTTP 503 с телом в OpenAI-формате ошибки + список попыток + Telegram-алерт |
| `.env` не найден / пуст | Остаётся только keyless-провайдер (`aihorde`). В лог пишется `❌ Ни одного доступного провайдера` |
| `AGENTS.md` не найден | **Файла нет в репозитории вообще**, хотя README ссылается на него 8 раз. Фактически протокол живёт в `config.yaml → agent.system_prompt` |
| Перезапуск во время запроса | Статистика и лимиты сбрасываются в БД при shutdown, восстанавливаются при старте |
| 100 одновременных запросов | **Не тестировалось.** Устранена главная проблема — SQLite на каждый запрос |
| Провайдер вернул не-JSON | `json.JSONDecodeError` перехватывается, провайдер помечается сбойным, берётся следующий |
| System prompt > 100K токенов | Consilium не обрезает. Управляется на стороне Hermes через компрессию по `usage.prompt_tokens` |
| Ключ со спецсимволами | Кавычки снимаются при парсинге `.env`; поддержан префикс `enc:` |

## 10. Рекомендации по модернизации

### Немедленно после деплоя

```bash
hermes doctor
```

В v0.19 эта команда показывает неизвестные корневые ключи конфига и
устаревшие поля. Это подтвердит, правильно ли Hermes принял секцию
`providers.custom`.

### Архитектура

`consilium_server.py` всё ещё знает о провайдерах слишком много: ветки
`if provider["name"] == "cloudflare"` и сборка payload для разных форматов
сидят в сервере. Правильное место — метод `build_payload()` в классе
провайдера. Тогда добавление провайдера = один файл без правок сервера.

Следующий шаг — `providers/registry.py` с автообнаружением по каталогу,
чтобы `ALL_PROVIDERS` не нужно было править руками.

### Оптимизация для VIM4 8 GB

- Главное уже сделано: SQLite больше не открывается на каждый запрос.
- `httpx.Limits(max_connections=150)` избыточен для 8 ГБ — достаточно 30–50.
- `uvicorn --workers 1` **обязателен**: in-memory состояние DPS и лимитов
  не разделяется между процессами.
- `logger.info(f'📥 Body: ...[:3000]')` на каждый запрос — заметный объём
  записи на eMMC. Стоит перевести на `DEBUG`.

### Отказоустойчивость

- Семантический кэш ответов: агент часто повторяет однотипные запросы,
  это экономит и лимиты, и время.
- Пул ключей на уровне провайдера с учётом DPS по каждому ключу отдельно,
  а не только по провайдеру.

### Мониторинг

- `/stats/providers` уже отдаёт DPS — стоит вывести на дашборд.
- Заголовки `X-Request-ID` / `X-Consilium-Provider` / `X-Consilium-Model`
  добавлены; сквозная трассировка до Hermes требует логирования на его стороне.
